One error language across layers and protocols.

Core paths

What developers should learn first

The public path is intentionally small: define reasons, enter the structured system, remap across layers, then render stable boundary outputs.

source_err(reason, detail)

Use when a raw error first enters orion-error, or when an upper layer creates a new semantic boundary and keeps the lower error as source.

conv_err()

Use when a lower StructError only needs reason remapping. It preserves the existing boundary instead of wrapping it.

to_err()

Use when a single reason enum value should immediately become a structured error.

with_source(...)

Attach raw standard errors or lower structured errors through the unified source API. Explicit source variants are lower-level tools.

Positioning

Why this crate exists

orion-error is not a prettier display layer. It is for codebases where errors become contracts between services, operators, clients, tests, and protocol adapters.

ToolBest forLimit
anyhowFast application error handling and type erasureNo stable domain identity or protocol exposure model
thiserrorLocal error enums and standard Error implementationsNo built-in context, snapshot, or governance workflow
color-eyreRich terminal diagnostics for humansDisplay-oriented, not a stable machine contract
orion-errorLarge Rust services with layered boundariesMore explicit because identities and exposure are part of the design
