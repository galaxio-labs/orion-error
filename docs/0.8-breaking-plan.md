# 0.8 Breaking Plan

更新时间：2026-04-24

本文档记录 `orion-error 0.8` 如果进入“不再兼容旧版本路径”的窗口，需要完成的收口工作。

目标不是现在立刻执行，而是把已经确认的 breaking 候选项、依赖原因和实施顺序固定下来，避免后续再次靠聊天记录回忆。

## 1. 当前已知不能直接删除、但属于兼容残留的 root 入口

以下名字目前仍保留在 crate root，原因不是“推荐主路径”，而是 derive 宏、示例或兼容导入仍依赖它们：

- `ErrorCategory`
- `ErrorCode`
- `ErrorIdentityProvider`
- `ErrorRenderer`
- `RenderMode`
- `TextDiagnosticRenderer`
- `TextReportRenderer`
- `Visibility`

## 2. 0.8 应完成的 API 收口

### 2.1 评估其余 root hidden 兼容入口是否整体移除

在 `DomainReason` 收口后，下面这批应重新评估是否还保留 root hidden 兼容：

- `ErrorCategory`
- `ErrorCode`
- `ErrorIdentityProvider`
- `ErrorRenderer`
- `RenderMode`
- `TextDiagnosticRenderer`
- `TextReportRenderer`
- `Visibility`

建议原则：

- 如果某个名字仍承担 derive 入口职责，就保留
- 如果只是“历史上放在 root 也能用”，则迁回分层模块

当前判断：

- `ErrorCode`
- `ErrorIdentityProvider`

大概率仍应保留，因为它们同时承担 derive 名字入口。

## 3. 0.8 应完成的 compat API 移除

以下接口曾属于 compat / deprecated 路径，已在 `0.8.0` 当前实现中移除：

- `ErrorOwe*` / `owe(...)`
- `ErrorWrap::err_wrap(...)`
- `WrapStructError::wrap(...)`
- `want(...)`
- `with(...)`

这里的原则已经固定：

- 新代码主路径只讲 `doing(...)`
- source 主路径使用 `with_source(...)` / `source(...)`

关于 source API：

- `with_source(...)` / `StructErrorBuilder::source(...)` 保留为主路径推荐入口
- `with_std_source(...)` / `with_struct_source(...)` / `source_std(...)` / `source_struct(...)`
  保留为少数需要显式表达 source 通道的进阶入口

## 4. 0.8 应完成的模块边界收口

### 4.1 重新评估 `advanced_prelude`

当前定位已经收敛为：

- 协议/schema 测试
- 迁移验证
- 大范围编译覆盖

0.8 应决定它是：

- 继续保留，但明确是测试/迁移工具箱
- 继续瘦身
- 或拆成更小的专用模块

## 5. 0.8 应完成的文档收口

需要同步更新：

- `README.md`
- `docs/README.md`
- `docs/tutorial.md`
- `docs/protocol-contract.md`
- `docs/stable-snapshot-schema.md`

需要明确写清：

- 哪些 root 名字是长期保留的正式入口
- 哪些 root 名字只是 derive 例外
- 哪些 compat 路径已被移除
- `advanced_prelude` 的最终定位

## 6. 建议执行顺序

推荐按下面顺序推进：

1. 先收 root hidden export
2. 再移除 compat API
3. 最后更新 README / tutorial / docs 索引

## 7. 当前结论

到 `0.8.0` 当前状态为止，可以确定：

- root hidden export 已经收窄到较小集合
- 大部分历史 hidden alias 已经删掉

因此，`0.8` 的 breaking 工作重点，不再是继续零碎删 API，而是：

- 一次性正式收口 root 与 compat 表面
