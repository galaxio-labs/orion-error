# 0.9 Design Plan

更新时间：2026-04-30

本文档记录 `orion-error` 从当前 `0.8.0` 状态继续提升到 `9+` 设计质量时，建议执行的重构计划。

目标不是继续做零散 API 删除，而是让：

- 主路径只有一套叙事
- 内部模型不再被 compat 投影字段主导
- `report` / `snapshot` / `exposure` 三层职责更干净
- 文档、derive、测试、导出面长期锁定一致

## 1. 当前基线

当前 `0.8.0` 已经完成的关键收口：

- root 只保留主路径运行时入口和 derive 宏入口
- `DomainReason` / trait 形态 `ErrorCode` / trait 形态 `ErrorIdentityProvider`
  已迁到 `reason::*`
- `owe(...)` / `err_wrap(...)` / `wrap(...)` / `want(...)` / `with(...)`
  等 compat API 已移除
- source 主路径已经固定为：
  - `with_source(...)`
  - `StructErrorBuilder::source(...)`
- `runtime` 的 source 观察面已下沉到 `runtime::source::*`
- `report` / `protocol` / `cli` 已拆成独立公开层：
  - `report::*` 只保留人类诊断与 redaction
  - `protocol::*` 负责 exposure / 协议投影
  - `cli::*` 负责 CLI 输出糖衣
- `dev::prelude` 已收窄为 snapshot/report/protocol 检查面
- `DiagnosticReport` 已完成第一轮瘦身：
  - projection accessor 已对外收回
  - `want/path/root_metadata/source_frames` 已不再属于公开 report API
  - 内部 projection 数据已从 `DiagnosticReport` 拆到 `ErrorProtocolSnapshot`
- `ErrorProtocolSnapshot::from_report_skeleton(...)` 已明确降为“协议骨架构造器”

当前仍影响设计上限的主要问题：

- `OperationContext` 内部仍保留 `target/path` 兼容投影痕迹
- 协议投影层可以进一步统一内部实现，降低重复映射逻辑
- `ErrorProtocolSnapshot` 的手工构造与内部 projection 组装仍有一定重复

## 2. 阶段 A：0.8.x 收尾锁定

目标：避免 API 与文档回漂。

### A.1 文档总校对

确保 README / tutorial / docs 索引只描述当前主路径：

- `into_as(...)`
- `err_conv()`
- `wrap_as(...)`
- `with_context(...)`
- `doing(...)` / `at(...)`
- `with_source(...)` / `source(...)`
- `report()`
- `snapshot().stable_export()`
- `exposure_snapshot(...)`

### A.2 导出面测试

锁定：

- root 不再暴露 `DomainReason`
- root 不再暴露 trait 形态 `ErrorCode`
- root 不再暴露 trait 形态 `ErrorIdentityProvider`
- derive 宏名仍可从 root 导入
- `conversion` / `reason` / `runtime` / `report` / `snapshot` / `interop`
  的边界不回退

### A.3 derive compile tests

重点锁：

- `#[derive(OrionError)]` 只依赖 `reason::*`
- `#[derive(ErrorCode)]` 与 `#[derive(ErrorIdentityProvider)]`
  的宏展开不再要求 root trait 路径

## 3. 阶段 B：0.8.x 小重构

目标：减少低层重复入口和继续膨胀的风险。

### B.1 收回 `attach_source(...)`

补充说明：

- 主路径推荐 `with_source(...)`
- `attach_source(...)` / `IntoSourcePayload` 只是底层自动分流实现

建议：

- 改为 `pub(crate)`
- 从 root / bridge 导出面移除相关实现细节
- 主文档只保留 `with_source(...)` / `source(...)`

当前状态：

- `attach_source(...)` 已不在主文档路径
- root compile-fail guard 已锁住误用
- `runtime::source::*` 当前只保留只读观察模型：
  - `SourcePayloadRef`
  - `SourcePayloadKind`
  - `SourceFrame`
- `SourcePayload` 已不再属于公开 observation surface
- `IntoSourcePayload` / `attach_source(...)` 已彻底收进 `src/core/error.rs`
  内部实现

### B.2 继续收 `dev::prelude`

原则：

- 只面向测试、迁移验证、协议/schema 检查
- 不再重新承担 bridge/runtime/reason 的宽导出角色

当前状态：

- 这一步已基本完成
- `dev::prelude` 当前只覆盖 snapshot/report/protocol 检查面
- 后续重点是维持回归锁，而不是继续扩张

### B.3 继续收 `types`

原则：

- 只保留真正的共享 carrier
- 避免重新膨胀成“第二个 root”

## 4. 阶段 C：0.9.0 核心模型重整

目标：这是把设计分拉到 `9+` 的关键阶段。

### C.1 重构 `OperationContext`

目标文件：

- `src/core/context.rs`

建议方向：

- 让内部主语义围绕：
  - `action`
  - `locator`
  - `path segments`
  - `fields`
  - `metadata`
  - `result`
  - `mod_path`
- `target` 改为投影值，不再是核心存储字段

原因：

- 当前 runtime API 已经转向 `doing(...)` / `at(...)`
- 兼容字段不应继续主导内部状态模型

当前状态：

- `normalized_path_segments()` 已明确以 `action` 为主、`target` 为 compat
  fallback
- compat `target` 已不再自动镜像写入原始 `path` 存储
- 当 `action == target` 时，冗余 compat `target` 已会被主动清理
- `path` 当前更接近稳定 segments，`target()` / `want` 更明确地退回兼容投影
- 后续重点是继续收口命名和投影边界，而不是继续保留历史双写语义

### C.2 重构 `DiagnosticReport`

目标文件：

- `src/core/report.rs`

建议方向：

- `DiagnosticReport` 只保留“人类诊断”必需数据
- 把兼容投影字段拆到更下游的 projection 输入对象

可以考虑引入内部中间对象，例如：

- `ReportProjectionParts`
- `ProtocolProjectionInput`

原因：

- 当前 `DiagnosticReport` 同时承担 display 信息和投影准备数据
- 内聚度还不够纯

当前状态：

- 这一步已基本完成
- `DiagnosticReport` 当前只保留 `reason/detail/position/context`
- protocol projection 数据已经下沉到 `ErrorProtocolSnapshot`
- 后续重点不再是继续瘦 `DiagnosticReport`，而是继续统一 protocol 侧组装逻辑

### C.3 兼容字段只在导出层生成

这些字段保留在 stable export / protocol projection 没问题：

- `want`
- `path`
- `target`

但它们不应继续主导 runtime / report 内部模型。

## 5. 阶段 D：0.9.0 协议投影实现统一

目标：让协议层实现关系更干净，不只是 API 看着整齐。

### D.1 统一 projection 生成路径

重点整理：

- `to_http_error_json()`
- `to_cli_error_json()`
- `to_log_error_json()`
- `to_rpc_error_json()`

要求：

- 共享同一组装逻辑
- 避免多处散落的字段映射和 message/visibility 判定

当前状态：

- `ErrorProtocolSnapshot` 当前已显式导出两类内部 view：
  - user-debug view
  - protocol JSON view
- 四种 `to_*_error_json()` 当前都由统一 JSON projection view 派生
- 后续如果继续推进，重点应是是否还需要进一步提炼共享字段策略，
  而不是再回到分散 helper 拼装

### D.2 固定三层职责

- `report()`：给人看
- `snapshot().stable_export()`：稳定机器导出
- `exposure_snapshot(...)`：协议输出输入面

### D.3 维持单向依赖

建议关系：

- runtime -> report
- runtime -> snapshot
- runtime + identity + report -> protocol

不要让 report 再反过来承担 protocol 核心职责。

## 6. 阶段 E：质量锁定

目标：避免半年后再次回漂。

### E.1 compile tests

锁定：

- derive compile-pass
- 分层导入 compile-pass
- root 已移除 trait/type compile-fail

### E.2 feature matrix

至少覆盖：

- default
- `serde`
- `serde_json`
- `tracing`
- `all-features`

### E.3 文档代码块可编译

重点锁：

- `README.md`
- `README.zh-CN.md`
- `docs/tutorial.md`
- `docs/reason-identity-guide.md`

### E.4 导出面回归测试

每次改 `src/lib.rs` 时都跑导出面测试，禁止 root / layered namespace 回漂。

## 7. 建议执行顺序

推荐按下面顺序推进：

1. `OperationContext` 内部模型瘦身
2. protocol projection 内部统一
3. compile tests + feature matrix + docs compile

## 8. 9+ 验收标准

满足下面这些，可以认为 `orion-error` 达到 `9+`：

- runtime 内部模型不再被 compat 字段支配
- `DiagnosticReport` 不再像兼容字段仓库
- 主路径叙事稳定且唯一
- `report` 与 protocol/exposure 不再共用一个公开出口
- 公开 API 面没有重复概念
- derive / docs / tests / exports 长期锁定，不再靠人工记忆维持一致

## 9. 当前重点

截至 2026-04-30，后续最值得投入的事项已经收敛到两类：

1. `OperationContext` 内部模型继续去 compat 化
2. protocol projection 内部实现进一步统一

source 相关历史实现面已经基本收口完成：

- `with_source(...)` / `builder.source(...)` 继续作为唯一主路径
- source 自动分流 helper 已退回内部
- `runtime::source::*` 只保留观察面，不再暴露写侧 carrier

不建议再把下面这些当作 0.9 的主要工作面：

- 再重复做一轮 `DiagnosticReport` 瘦身
- 再把 `dev::prelude` 当作大块重构目标
- 继续从公开面删一些低收益 accessor 作为主要进展

## 10. 不建议的做法

下面这些做法不建议作为 0.9 的主要方向：

- 重新把 root surface 扩回“大而全”导出
- 再引入第二套 source 主路径
- 继续让 compat 投影字段主导 runtime 语义建模
- 把 `with_source(...)` 降回 compat-only

当前更合理的方向是：

- 主路径继续保持简洁
- 低层能力缩回内部或进阶层
- 内部模型彻底摆脱历史包袱
