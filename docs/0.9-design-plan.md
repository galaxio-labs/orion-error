# 0.9 Design Plan

更新时间：2026-04-29

本文档记录 `orion-error` 从当前 `0.8.0` 状态继续提升到 `9+` 设计质量时，建议执行的重构计划。

目标不是继续做零散 API 删除，而是让：

- 主路径只有一套叙事
- 内部模型不再被 compat 投影字段主导
- `report` / `snapshot` / `exposure` 三层职责更干净
- 文档、derive、测试、导出面长期锁定一致

## 1. 当前基线

当前 `0.8.0` 已经完成的关键收口：

- root 只保留主路径运行时入口和 derive 宏入口
- `DomainReason` / trait 形态 `ErrorCode` / trait 形态 `ErrorIdentityProvider`
  已迁到 `reason::*`
- `owe(...)` / `err_wrap(...)` / `wrap(...)` / `want(...)` / `with(...)`
  等 compat API 已移除
- source 主路径已经固定为：
  - `with_source(...)`
  - `StructErrorBuilder::source(...)`

当前仍影响设计上限的主要问题：

- `OperationContext` 内部仍保留 `target/path` 兼容投影痕迹
- `DiagnosticReport` 仍承载过多 projection 相关字段
- `advanced_prelude` 虽已收窄，但仍需长期约束
- 协议投影层可以进一步统一内部实现，降低重复映射逻辑

## 2. 阶段 A：0.8.x 收尾锁定

目标：避免 API 与文档回漂。

### A.1 文档总校对

确保 README / tutorial / docs 索引只描述当前主路径：

- `into_as(...)`
- `err_conv()`
- `wrap_as(...)`
- `with_context(...)`
- `doing(...)` / `at(...)`
- `with_source(...)` / `source(...)`
- `report()`
- `snapshot().stable_export()`
- `exposure_snapshot(...)`

### A.2 导出面测试

锁定：

- root 不再暴露 `DomainReason`
- root 不再暴露 trait 形态 `ErrorCode`
- root 不再暴露 trait 形态 `ErrorIdentityProvider`
- derive 宏名仍可从 root 导入
- `conversion` / `reason` / `runtime` / `report` / `snapshot` / `bridge`
  的边界不回退

### A.3 derive compile tests

重点锁：

- `#[derive(OrionError)]` 只依赖 `reason::*`
- `#[derive(ErrorCode)]` 与 `#[derive(ErrorIdentityProvider)]`
  的宏展开不再要求 root trait 路径

## 3. 阶段 B：0.8.x 小重构

目标：减少低层重复入口和继续膨胀的风险。

### B.1 收回 `attach_source(...)`

当前状态：

- 主路径推荐 `with_source(...)`
- `attach_source(...)` / `IntoSourcePayload` 只是底层自动分流实现

建议：

- 改为 `pub(crate)`
- 从 root / bridge 导出面移除相关实现细节
- 主文档只保留 `with_source(...)` / `source(...)`

### B.2 继续收 `advanced_prelude`

原则：

- 只面向测试、迁移验证、协议/schema 检查
- 不再重新承担 bridge/runtime/reason 的宽导出角色

### B.3 继续收 `types`

原则：

- 只保留真正的共享 carrier
- 避免重新膨胀成“第二个 root”

## 4. 阶段 C：0.9.0 核心模型重整

目标：这是把设计分拉到 `9+` 的关键阶段。

### C.1 重构 `OperationContext`

目标文件：

- `src/core/context.rs`

建议方向：

- 让内部主语义围绕：
  - `action`
  - `locator`
  - `path segments`
  - `fields`
  - `metadata`
  - `result`
  - `mod_path`
- `target` 改为投影值，不再是核心存储字段

原因：

- 当前 runtime API 已经转向 `doing(...)` / `at(...)`
- 兼容字段不应继续主导内部状态模型

### C.2 重构 `DiagnosticReport`

目标文件：

- `src/core/report.rs`

建议方向：

- `DiagnosticReport` 只保留“人类诊断”必需数据
- 把兼容投影字段拆到更下游的 projection 输入对象

可以考虑引入内部中间对象，例如：

- `ReportProjectionParts`
- `ProtocolProjectionInput`

原因：

- 当前 `DiagnosticReport` 同时承担 display 信息和投影准备数据
- 内聚度还不够纯

### C.3 兼容字段只在导出层生成

这些字段保留在 stable export / protocol projection 没问题：

- `want`
- `path`
- `target`

但它们不应继续主导 runtime / report 内部模型。

## 5. 阶段 D：0.9.0 协议投影实现统一

目标：让协议层实现关系更干净，不只是 API 看着整齐。

### D.1 统一 projection 生成路径

重点整理：

- `to_http_error_json()`
- `to_cli_error_json()`
- `to_log_error_json()`
- `to_rpc_error_json()`

要求：

- 共享同一组装逻辑
- 避免多处散落的字段映射和 message/visibility 判定

### D.2 固定三层职责

- `report()`：给人看
- `snapshot().stable_export()`：稳定机器导出
- `exposure_snapshot(...)`：协议输出输入面

### D.3 维持单向依赖

建议关系：

- runtime -> report
- runtime -> snapshot
- runtime + identity + report -> protocol

不要让 report 再反过来承担 protocol 核心职责。

## 6. 阶段 E：质量锁定

目标：避免半年后再次回漂。

### E.1 compile tests

锁定：

- derive compile-pass
- 分层导入 compile-pass
- root 已移除 trait/type compile-fail

### E.2 feature matrix

至少覆盖：

- default
- `serde`
- `serde_json`
- `tracing`
- `all-features`

### E.3 文档代码块可编译

重点锁：

- `README.md`
- `README.zh-CN.md`
- `docs/tutorial.md`
- `docs/reason-identity-guide.md`

### E.4 导出面回归测试

每次改 `src/lib.rs` 时都跑导出面测试，禁止 root / layered namespace 回漂。

## 7. 建议执行顺序

推荐按下面顺序推进：

1. `OperationContext` 内部模型瘦身
2. `DiagnosticReport` 瘦身
3. `attach_source` / `advanced_prelude` 收口
4. protocol projection 内部统一
5. compile tests + feature matrix + docs compile

## 8. 9+ 验收标准

满足下面这些，可以认为 `orion-error` 达到 `9+`：

- runtime 内部模型不再被 compat 字段支配
- `DiagnosticReport` 不再像兼容字段仓库
- 主路径叙事稳定且唯一
- 公开 API 面没有重复概念
- derive / docs / tests / exports 长期锁定，不再靠人工记忆维持一致

## 9. 不建议的做法

下面这些做法不建议作为 0.9 的主要方向：

- 重新把 root surface 扩回“大而全”导出
- 再引入第二套 source 主路径
- 继续让 compat 投影字段主导 runtime 语义建模
- 把 `with_source(...)` 降回 compat-only

当前更合理的方向是：

- 主路径继续保持简洁
- 低层能力缩回内部或进阶层
- 内部模型彻底摆脱历史包袱
